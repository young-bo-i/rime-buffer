# Buffer UI Notes

## Intent

The buffer is a native IME-layer staging area. It is not an AX/paste workaround and it is not part of Rime composition. Rime commits become blocks; blocks remain visibly staged until they are removed or flushed to the focused client through the normal delivery path.

## Current Shape

- `BufferModel` is an ordered block store, not an append-only queue anymore. A `Block` has stable `id`, `text`, `origin`, and `createdAt`. Rime commits establish block boundaries; accepted external text can also enter with provenance.
- Blocks can be inserted at `insertionIndex` or removed through explicit model operations. Workbench chips are passive and do not open a block editor. Live blocks are pending by definition because successful delivery consumes them.
- `BufferWindowController` owns the independent workbench. It is a borderless, nonactivating `NSPanel`, not a bottom-of-screen `BufferSurface`. Ordinary mode is a 44pt collapsed single-line rail and a 78pt expanded panel. Generated layouts use one source plus one to three independently scrolling target rows: collapsed heights are 78/109/140pt and expanded heights are 112/143/174pt. Every transition grows upward while the bottom edge and Rime-candidate anchor stay fixed. Width may be adjusted, while the former tall preview layout is gone. Only the dedicated compact 22pt drag-handle icon may initiate window movement, and it exposes the pointing-hand cursor on hover; the panel background and other controls must not drag. Visibility, frame, controls-expanded state, and pin-to-all-spaces are persisted in UserDefaults.
- The ordinary collapsed main row is exactly: drag handle, expand/collapse affordance, `BufferInlineView`, and the explicit send-next paper plane. The upward shelf contains only a target-free status message, a compact buffer-plugin selector plus the selected plugin's actions, refresh/reset, and close. The selector lists every `.bufferAction` plugin plus “no plugin”; selecting a disabled plugin enables it and atomically replaces the unique owner. Refresh/reset never clears staged text: for an external action plugin it cancels stale/current work, clears the current failure, and forces context/status detection; for a derived workspace it preserves source text, invalidates the current generation, and restarts its current operation. Pin, move, and the underlying buffer enable/disable control remain available from Settings/the input-source menu, not the workbench. There is no block editor, in-panel capture switch, manual privacy-eye, history/restore, or clear/undo control. The workbench does not embed a Rime candidate projection or a full-text preview; stream alternatives are derived target rows, not Rime candidates. An ordinary close resolves current composition if safe, pauses capture, clears transient loading/error state, preserves staged blocks, and hides.
- Apple Translation, the single AI 生成 plugin, and 意识流输入 use independently scrolling source/target rows. Translation, AI 生成, and a one-candidate stream result have one source plus one target row. Two or three stream alternatives add one or two target rows below; each candidate owns one stable row, and host semantic segmentation creates multiple delivery chips only inside that row. Rail roles are SF Symbols with accessibility labels, never the visible Chinese markers “原/译/答/拼/文”. AI 生成 uses the independently selected Codex CLI, Claude Code CLI, or 通用 Open API（OpenAI 兼容）connector. 意识流输入 instead owns focus-bound raw full-pinyin outside BufferModel/Rime and always uses the user's saved OpenAI-compatible configuration directly, independent of ordinary connector selection; the current configured model is `deepseek-v4-flash`. The drag handle and disclosure affordance align directly with the upper source row; the primary action aligns with the lowest target row.
- AI 生成 is explicitly generated, never automatic-on-type. It has no separate expanded-shelf Generate button: the right-side primary action is a disabled AI icon with no source, an enabled AI icon when generation can be requested, a spinner while running, and a paper plane when complete target blocks are ready. Return uses the same state machine; request/running/disabled own the entire press on keyDown, while ready retains tap `sendNext` / 1.2-second hold `sendAll`. A generation freezes current source text and source block identities; source remains through generation and partial delivery, and is consumed only after every target block has been delivered successfully.
- 意识流输入 is the intentional automatic exception: after the 220/800 ms debounce/max-wait gates, the model returns one to three probability-ordered, complete, mutually exclusive guesses. Each letter resets only the 220 ms debounce; the 800 ms deadline established for the current typing burst does not move, so continuous typing still refreshes. Space records one real normalized ASCII short-sentence boundary in raw, renders only as `·` on the source rail, and immediately requests the current complete raw snapshot through the same bounded scheduler; leading/repeated spaces do not create work. Continuing after Space still receives the normal trailing debounce request for the newest complete raw. Other punctuation remains consumed and ignored. It should return one result for clear input and add alternatives for meaningful ambiguity; when local hints contain more than one reading, the payload sets `minimumGuessCount=2`. Never split one intended sentence across alternatives. Digits `1`–`3` and plain Up/Down change the unconfirmed selection; modified arrows remain host commands. Return keyDown or the paper plane atomically confirms the selected row and deletes its peers before any insertion. The same Return tap then sends the next semantic block and a 1.2-second hold sends every remaining block of that selected answer; alternatives are never concatenated.
- Stream inference uses bounded make-before-break, never unbounded fan-out. At most an older visual producer and one newer challenger may overlap. Typing a newer raw value immediately revokes every old result/delivery lease but does not cancel issued requests on each key. At a debounce, max-wait, Space, or Return boundary, start the newest complete snapshot when a slot is free; keep the prior visual producer until the challenger emits its first validated nonempty snapshot or reaches a terminal result, then tombstone/cancel the older request. If both slots are occupied, keep only one latest-only pending marker; when a slot opens it captures the then-current complete raw instead of replaying an obsolete queued snapshot. Provisional output and the baseline are display-only: never include either in a later prompt, show either as selected while waiting/running, or make either deliverable.
- Every 意识流 request carries the complete raw-pinyin snapshot frozen at its request boundary and asks for a fresh global interpretation; never infer only the newly typed suffix or concatenate separately inferred regions. A bounded local syllable lattice supplies up to three lossless advisory boundary readings, renders every raw Space boundary as ` | ` in hints, and preserves uncertain English/typo spans; above 512 bytes it omits hints rather than duplicating a large raw value. It never generates Chinese or partitions model inference. Space-separated nonempty raw clauses establish a minimum delivery-block target; when the model omits punctuation, split further only at safe boundaries and never cut an English word, URL, code span, number, or quotation merely to reach that target. Reuse stable candidate-slot identities and reconcile each new partial in place from its common prefix; retain the inert baseline while a divergent replacement is still shorter than four readable characters, then replace in place. The current-raw final response is authoritative and must exactly replace its generation's slots; no baseline tail, prior generation, or separately inferred region may enter a ready/deliverable block. Confirmation prunes peer alternatives immediately, while raw survives partial delivery and clears only after the selected answer's final block succeeds; typing a letter or Space after the first delivered block deliberately drops the unsent tail and starts a fresh raw value.
- Codex CLI and Claude Code CLI are local process launches, not local inference: clicking Generate sends the current buffer text through the service authenticated by the installed CLI. Except for an explicit fail-closed `RIMEBUFFER_CODEX_PATH` override, Codex discovery must prefer ChatGPT.app's bundled executable and then continue to the first compatible Homebrew/PATH candidate; its ChatGPT credential remains isolated under `~/Library/RimeBuffer/ai/codex-home` and never comes from `~/.codex`. Claude readiness must be derived only from the official `claude auth status --json` `loggedIn` result; Settings may launch/cancel `claude auth login --claudeai`, but RimeBuffer must not read Claude credential files or pass `CLAUDE_CODE_OAUTH_TOKEN`, `CLAUDE_CONFIG_DIR`, or ambient API keys to the child. Version/auth probes run off the IMK main thread with stale-while-revalidate caching; Generate verifies the cached executable's stat fingerprint before source text is prepared. Both providers fail closed before source text leaves RimeBuffer when no executable is inside the verified tool-surface allowlist (currently Codex `0.144.1`/`0.145.0-alpha.18`, Claude Code `2.1.211`/`2.1.215`). The 通用 Open API（OpenAI 兼容）connector is configured with Base URL, model, and API key under Settings › Connectors › AI Model. 意识流的 `.alternativeGuesses` request explicitly sends `thinking: {type: disabled}`, JSON-object response mode, `max_tokens: 1024`, and low temperature; ordinary AI generation does not inherit these provider-specific controls. Its private JSON is `~/Library/RimeBuffer/ai/openai-compatible.json`, mode 0600; the key must not enter UserDefaults or logs. The `ai` directory is product-owned state outside the app bundle and must survive development reseeding, pkg overwrite installs, and in-app updates so users do not have to re-enter the configuration.
- `Command+Shift+B` is the process-global workbench toggle. When hidden it invokes the open-and-resume behavior, bringing the panel to the active Space and re-enabling capture. When visible it invokes the ordinary close-and-pause behavior, resolving composition safely, preserving blocks, stopping transient work, and pausing capture. The registered hot key is consumed and must not leak `B` to the host.
- `BufferWindowGeometry.clampedFrame` restores off-screen frames, chooses the screen with greatest intersection, clamps width to the selected screen, and normalizes ordinary 44/78pt plus derived 78/109/140pt collapsed and 112/143/174pt expanded heights. Every height transition preserves the window's bottom edge, so the Rime-candidate anchor does not jump. Runtime min/max sizes are resynchronized per screen after move/resize.
- `RimeBufferController.drainCommit` remains the routing point: buffer off means direct `Delivery.insert`; buffer on means append the commit to `BufferModel`. Preedit remains in marked text and the regular `CandidateWindow`, never in a stored block.
- While buffer mode is on, plain or Shift-modified Return and Backspace are always consumed by the input method, including engine-failure or untrusted-focus paths. A Return press with pending Rime/chord composition or a stream raw value that is not ready only settles/forces that source and suppresses the rest of the same physical press. A ready stream final instead confirms and prunes alternatives on keyDown, then uses that same physical tap/hold gesture. Otherwise a tap requests `sendNext` and a hold of about 1.2 seconds requests `sendAll`; Backspace edits active Rime/chord state or removes a buffer block only when precise focus is trusted. Neither key may fall through to the host text field.
- Exact Control/Command+A and Control/Command+V edit the workbench source for ordinary buffering and every plugin. Ordinary, Translation, AI, and Action/Marine modes select all `BufferModel` source blocks; paste inserts at the block caret or replaces the complete selection while preserving exact concatenated text before semantic segmentation. Stream mode selects only raw source, never candidate rows. Its paste accepts only ASCII letters plus whitespace, lowercases letters, normalizes whitespace runs to real ASCII Space boundaries, and immediately requests the complete new raw. Any other character or a normalized total above 16 KiB rejects the entire stream paste without changing raw, selection, alternatives, generation, or provider work. Ordinary clipboard text must be nonempty, NUL-free, and at most 1 MiB. Extra Shift/Option, Control+Command, and own-window shortcuts remain native. Secure input passes the shortcut through before any pasteboard access; an untrusted external focus consumes it without reading. Paste revalidates the same exact lease both before the read and after a potentially lazy pasteboard provider returns.
- A standalone Shift tap shorter than 500 ms keeps the schema's existing ASCII-toggle behavior. The controller defers the Rime Shift pair until physical release and replays the saved left/right keysym only after proving the gesture was a same-session/schema standalone tap. If Shift participates in another key (including controller-owned direct punctuation, buffer Return, stream capture, a shortcut, a pre-held modifier, or the other Shift), remains held for at least 500 ms, or crosses focus, neither mode-switch event is sent. This prevents `commit_code` side effects as well as preserving the prior Chinese/English mode.
- While an exact external buffer lease owns those controls, `CompositionSession` keeps one invisible U+200B marked-text guard alive even when Rime is idle. It reasserts that guard specifically on an idle Return keyDown before the tap/hold timer begins. This prevents Chromium/ProseMirror hosts from submitting the raw Return before/around IMK's handled result, including when a host silently ended an earlier marked session, without changing the existing short-press/long-press decision. Guard lifetime is independent from semantic Rime composition: idle guard means `composition.composing == false`, `FocusLease.compositionActive == false`, and `markedRangeReliable == false`. Every Return action synchronously checks secure input first: a secure press is consumed without composition settlement, U+200B reassertion, AI generation, or delivery. Transient external content otherwise uses the idle guard but keeps real Rime preedit inline; own-window focus, focus loss, or buffer deactivation removes the guard.
- `BufferDeliveryCoordinator` is the only component allowed to turn staged blocks into `Delivery.insert` calls. Keyboard delivery passes the `FocusToken` captured on Return keyDown; the explicit paper-plane control at the right end of the main row is also a current-focus `sendNext` action. The window never retains or calls an `IMKTextInput` directly.
- Switching the current buffer-plugin owner cancels in-flight work but is not an authority-revocation event for already completed Action Plugin blocks. A completed Marine block must retain the runtime binding, action, context, and focus validation minted when it was generated; disabling, uninstalling, or upgrading that external plugin still revokes those grants.
- A send starts from `InputFocusCoordinator.liveTarget()`. The lease must have the current and expected `FocusToken`, an external target, trusted delivery state, live controller/client, a `controller.client()` whose object identity still matches the lease, matching client bundle, and a frontmost application whose process PID (plus bundle when available) matches the lease. There is no recent-controller or last-client fallback.
- If composition is active, the coordinator may explicitly resolve it, then must reacquire the same expected token. It revalidates that target, composition state, and secure input before every block, stopping immediately if focus changes.
- A successful `Delivery.insert` call is only an accepted attempt, not a destination ACK, but the current product contract consumes that accepted block from the live buffer immediately. A failure stops the operation; the failed block and every not-yet-sent block stay in place.
- Accepted deliveries are consumed from live `blocks` without retaining a plaintext delivery history. There is no user-facing clear/undo path; automatic privacy discard remains an irreversible safety operation.
- Automatic flushing and timer-based deletion remain disabled. Return tap/hold and the paper-plane button are deliberate user actions; the paper plane is one-block only, while an accepted `insertText` attempt consumes the corresponding live block synchronously even though the target app does not provide a stronger acknowledgement.

## Display Requirements

- Show block boundaries because they are meaningful Rime commit boundaries.
- Keep current model order and the insertion point visually clear; explicit insertion means it is not always strict creation-time order.
- Show provenance for non-Rime blocks. The live rail contains pending blocks only; do not render sent checks or other sent-state decoration.
- Make long block text inspectable with a tooltip while keeping each individual rail row single-line.
- Keep passive text blocks dense without shrinking the 12pt text: use a 3pt gap between blocks, 4pt horizontal and 1pt vertical content insets, a 20pt block height, and 5pt rail-side insets. Apply the same density to ordinary, preedit, message, and translation blocks while preserving provenance and independent scrolling.
- Keep stream-input alternatives visually continuous across whole-raw generations: retain provisional old-job text and the handoff snapshot only as a visibly stale, noninteractive baseline until same-index partials can update stable slots in place. Never blank the target rail merely because raw changed or a newer request started. Waiting/running output has no selected treatment and no delivery lease; only a final whose raw/job/generation is still current may become selected and sendable.
- Keep first-click controls working inside nonactivating panels by using controls that accept first mouse.
- Give every enabled workbench button and popup an explicit hover/pressed surface plus a pointing-hand cursor using tracking that remains active in the nonactivating panel. Disabled controls keep the arrow and no hover surface. The drag handle uses the same visible feedback and remains the only draggable region; passive buffer chips/status text keep the arrow and never imply clickability.
- Keep the plugin selector compact and compressible. Store a complete domain-qualified `PluginKey` in each item rather than deriving identity from menu indices or raw IDs; rebuild it on the next main-loop turn after registry/selection notifications so AppKit never loses the item currently dispatching its action.
- Keep the expanded shelf's status column and plugin selector pinned to one leading position. A permanent flexible spacer after plugin actions must absorb status/action width changes so plugin controls only grow rightward while refresh and close remain fixed at the trailing edge.
- Show Return hold progress as a 2px accent line along the bottom of the always-visible rail without changing panel height; clear it on release, cancellation, focus change, or secure-input shielding.
- Show Rime composition candidates through the regular `CandidateWindow`, with the same theme, metrics, paging, and token-owned actions as normal input. In buffer mode its anchor is the workbench bottom edge, so the panel opens immediately below it (or flips above only when screen bounds require it); never duplicate Rime candidate state or project it into the workbench. Stream alternatives are derived target rows and do not use `CandidateWindow`.
- Treat an ordered panel on another macOS Space as not currently visible. An explicit Show action may re-order the unpinned strip onto the active Space.
- Keep ordinary mode at 44/78pt and derived one/two/three-target layouts at collapsed 78/109/140pt and expanded 112/143/174pt, with bounded width. Long text should truncate or scroll instead of resizing the panel off screen. The bottom edge must remain stationary across expansion and target-row changes so the Rime-candidate anchor stays stable. Geometry must still fit visible frames narrower than the normal minimum.
- Inset the rounded chrome inside a transparent window margin, fill it with the fixed day/night workbench surface token so HUD backdrop sampling cannot reduce text contrast, and draw the border as a backing-scale hairline inside its path. Do not center a layer border on the window bounds, where macOS clips it into fuzzy corner/edge fragments.
- Keep queued content visible even if buffer mode is switched off; do not hide unsent text.
- Keep an empty state visible when buffer mode is on but no blocks exist.
- Shield staged text plus candidate/delivery rails while secure input is active. Hide the workbench and revoke its target while the desktop session is inactive, locked, or asleep. There is no separate manual privacy-shield state.

## Safe Extension Points

- Add read-only presentation properties to `BufferModel`, such as grouped counts, source summaries, ages, or pending-delivery summaries. Keep mutation in explicit model methods so identity, provenance, order, and notifications stay coherent.
- Add richer single-line presentation to `BufferWindowController`/`BufferInlineView`, but route send actions through `BufferDeliveryCoordinator` and candidate actions through the token-owned `CandidateWindow` state machine.
- Add transformations by deliberately creating a new derived block with explicit provenance; do not silently mutate a passive workbench chip.
- Add a new generated two-rail plugin through a `BufferDeliveryContentSource`-compatible workspace: freeze source text/block IDs, invalidate stale generations, keep streaming block identity stable, and consume source only after the last target block succeeds.
- Add visibility, placement, or appearance settings through `SettingsWindow` and UserDefaults. Reuse `BufferWindowGeometry.clampedFrame` for restoration and add tiny-screen cases whenever geometry rules change.
- Evolve focus handling through the pure `FocusTargetRules`, `FocusEventRules`, and `FocusActivationRules` predicates. Any relaxation must retain exact-token ownership, event ordering, lifecycle attribution, external-target filtering, lease/client plus `controller.client()` identity checks, and frontmost bundle/PID checks, with matching `buffer-window-smoke` cases. Pending chord callbacks must be isolated before a lease is suspended or displaced.
- Add another delivery destination only behind an explicit routing layer. Local text insertion must still end at `Delivery.insert`, and a multi-block operation must stop rather than switch destinations when the original token becomes stale.
- Add persistence for buffer text only as an explicit privacy/product decision. Current content is intentionally process-local, and delivery history is intentionally not retained.
- Do not add automatic or delayed delivery outside the explicit Return gesture and paper-plane actions. A keyboard gesture must retain the keyDown token, deliberately settle any intervening composition without sending in the same press, and revalidate that exact token immediately before every insertion.

## Do Not Do

- Do not write buffer content directly to clients from `BufferWindowController` or `BufferInlineView`.
- Do not restore `active ?? recent`, `lastClient`, or bundle-only target fallbacks. The frontmost PID check is required because an app can relaunch with the same bundle identifier.
- Do not flush through an unresolved composition. Only `BufferDeliveryCoordinator` may resolve it for a user-initiated send, after which it must reacquire the same token.
- Do not let Return or Backspace escape to the host while buffer mode is enabled, even when Rime is unavailable or the focus lease is untrusted. Return delivery may occur only through the tap/hold state machine; repeats, late keyUp/command callbacks, focus changes, and composition-settling presses must never create an extra send.
- Do not implement timer-based deletion or automatic delivery. Consuming a block is part of the synchronous user-initiated successful delivery transaction; failure must leave that block live.
- Do not treat “CLI runs locally” as a local-inference privacy guarantee, put source text in CLI argv, invoke a shell, log provider output, or persist an API key in UserDefaults.
- Do not clear completed Action Plugin delivery bindings merely because the workbench owner changes; conversely, do not preserve them across disable, uninstall, upgrade, or runtime-identity failure.
- Do not make the passive panel key by default.
- Do not reintroduce diff/reconcile logic from old `buffer-bar`; commit boundaries are already known.
- Do not put preedit into the buffer. Composition preview belongs to marked text and/or candidate window.
- Do not persist buffer text/history accidentally while adding window preferences; those have a different privacy lifetime.
